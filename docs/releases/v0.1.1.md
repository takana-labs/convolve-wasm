# @takana-labs/convolve-wasm v0.1.1

**Status:** Candidate. Publication is blocked until the physical Android gate is recorded as Pass.

## Mobile-safe rejection

- Adds a checked post-decode, pre-worker peak-memory plan.
- Uses device-class browser budgets from 64 MiB through 384 MiB.
- Rejects risky work with `INPUT_TOO_LARGE` before worker creation.
- Includes estimated bytes, budget, input/output/FFT frame counts, reverse/crossfade state, beat-pan state, and reported device memory in error details.
- Keeps the independent 256 MiB Rust/WASM guard.
- Shows actionable estimated-versus-available memory guidance in the demo.

The full initial estimate is `E + 3D + F + R + X + P + 4W + 32 MiB`, using the definitions in the architecture document.

## Local application lifecycle

The repository now owns hidden start, stop, restart, status, URL, readiness, bounded log-tail, and diagnostics commands. Logs append under `app_work/runtime/`.

## Compatibility

Successful processing remains full-engine stereo 48 kHz signed PCM24 WAV. Progress stages, result shape, reverse semantics, beat detection behavior, and the independent WASM guard are unchanged.

This release establishes safe rejection on Android Chrome; it does not yet claim completion of large mobile renders. iOS remains best effort.

## Release gate

The exact private pair is identified only by hashes in the [mobile safe-rejection evidence](../testing/2026-07-17-mobile-safe-rejection.md). The release-candidate workflow refuses to package v0.1.1 until that record contains a physical Android Pass.

v0.1.2 lower-memory full FFT work and v0.2.0 bounded convolution remain blocked pending review of this milestone.
